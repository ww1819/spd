# 高值耗材即入即出 — 业务需求规格（草案）

> **文档状态**：待评审 / 可修正  
> **适用范围**：衡水三院等高值计费场景（与 HIS 计费镜像、科室高值消耗单联动）  
> **说明**：本文档仅整理业务描述与完整性评估，**不涉及代码实现**。

---

## 1. 背景与目标

### 1.1 业务背景

高值耗材的 **HIS 计费 / 退费** 需由科室 **手工** 完成高值消耗单、反消耗单（扫码院内码，扣减/回补高值科室库存）。

在此基础上，系统需按高值消耗明细 **自动生成低值即入即出仓单**，用于补齐仓库账务链路：

| 消耗记录数量符号 | 含义           | 自动生成低值单据                         |
|------------------|----------------|------------------------------------------|
| 正数             | 正向消耗       | **入库单（101）** + **出库单（201）**   |
| 负数             | 退消耗（退费） | **退库单（401）** + **退货单（301）**   |

单据生成后 **立即自动审核**，并生成仓库库存、科室库存流水；**仓库与科室库存余额均为 0**（即入即出，不在科室留存量）。

### 1.2 核心目标

1. 每条高值消耗明细可 **完整追溯** 至：低值单据明细 → 仓库/科室库存明细 → 院内码 → 产品主/辅条码。
2. 计费行与退费行存在关联时，**批次与库存 ID 必须一一对应**，避免退库/退货使用与入库/出库不一致的批次导致负库存。
3. 支持 **纠错冲销**：备错仓库、供应商、科室等已即入即出记录，可回退至「未消耗 / 可重新处理」状态。

### 1.3 与现有功能关系

| 现有能力                         | 本功能关系                                   |
|----------------------------------|----------------------------------------------|
| 患者收费查询 · 高值扫码消耗      | 产生高值消耗单，是本功能的数据来源之一       |
| `his_mirror_consume_link`        | 已有镜像↔消耗明细关联，需扩展至低值仓单/库存 |
| 低值计费冲销（`writeOffLowValue`） | 仅低值；**高值即入即出冲销为独立能力**       |
| 高值退费返还                     | 产生反消耗单，触发负数即入即出               |

- 用户 **勾选** 待处理记录；

- 点击 **审核**（即「执行即入即出」）批量生成低值单据。

#### 3.2.2 已审核 Tab 操作

- 展示已生成的低值单据号、审核时间等（字段待 UI 设计）；
- 支持 **纠错冲销**（见第 6 章）。

### 3.3 列表数据来源

列表行粒度为：**高值消耗单明细级**，且仅展示“同一科室 + 同一供应商”的消耗明细记录（用于同科室同供应商合并成一张即入即出单据）。目标仓库由用户在审核弹窗中选择。

需明确一行记录至少包含：

- 高值消耗主表：单号、消耗日期、科室、仓库（来源仓）、备注
- 高值消耗明细：数量（正/负）、院内码、主条码、辅条码、供应商、产品、单价、批次、效期等
- 计费/退费镜像：镜像行 ID、HIS 收费明细 ID、收费项目、患者/科室快照（是否展示）
- 即入即出状态：未做 / 已完成 / 已冲销
- 低值单据引用：101/201 或 401/301 单号（已审核 Tab）

## 4. 即入即出审核（生成低值单据）

### 4.1 触发与前置校验

| 规则 | 说明 |
|------|------|
| 操作入口 | 未审核 Tab，勾选记录后点击「审核」 |
| 供应商必填 | 消耗记录内 **无供应商则不允许** 生成即入即出 |
| 仓库 | 用户点击「审核」后，在弹窗中 **手工选择目标仓库**（当前不自动推断归属） |
| 科室 | 取 **高值消耗单主表科室** |
| 数量方向 | 正数 → 101 + 201；负数 → 401 + 301 |
| 明细粒度 | **按消耗单明细** 生成低值明细（逐条生成明细，保证逐条追溯）；单据主表按“同科室 + 同供应商”合并成 **一单**（见 4.2/4.3）。 |

> **定稿**：一次审核合并规则为“同一科室 + 同一供应商”的勾选记录生成 **一张** 即入即出单据（按数量正负分别走 101+201 或 401+301；明细逐条生成）。

> **定稿补充**：一次勾选审核 **禁止混合正数与负数**。若勾选集内同时存在正数与负数，应提示用户拆分为两次操作：正数生成 101+201，负数生成 401+301（严禁合并成同一张单据），避免来源计费/退费数据不一致导致链路混乱。

### 4.2 单据头规则

| 字段     | 取值规则 |
|----------|----------|
| 仓库     | 用户选择的仓库 |
| 供应商   | 消耗记录明细内的供应商 |
| 科室     | 高值消耗单主表科室 |
| 单据来源 | 标记为「即入即出生成」（主表备注 + 业务类型字段） |
| 审核     | 生成后 **立即自动审核** |

### 4.3 单据类型与库存影响

**正向（数量 > 0）**

1. 生成 **入库单 101** → **审核** → 生成 **仓库库存**（`stk_inventory`）
2. 再生成 **出库单 201** → **审核** → 生成 **科室库存**（`stk_dep_inventory`），科室库存采用“**直接生成 qty=0 的库存行**”方式实现（不再额外生成一遍低值消耗数据去扣减科室库存，降低代码复杂度）

**逆向（数量 < 0，退消耗）**

1. 生成 **退库单 401** → 审核 → 从科室退回仓库（科室净 0）
2. 生成 **退货单 301** → 审核 → 从仓库退供应商

**库存余额要求**

- 即入即出完成后：**仓库库存余额 = 0，科室库存余额 = 0**（允许有流水、无结存）。
- 单据明细与库存明细必须可追溯到：**高值消耗明细 ID、院内码、产品主条码、产品辅条码**（快照写入，禁止仅靠 join 反查）。

### 4.4 低值单据与消耗单关联（强制）

**低值单据明细** 必须记录：

- 高值消耗单 **主表 ID**
- 高值消耗单 **单号**
- 高值消耗单 **明细 ID**

**低值单据主表** 备注示例：

> 即入即出生成；来源高值消耗单号 xxx；镜像行 xxx（可选）

**批次与库存 ID 匹配（强制）**

- 101 入库产生的 **批次 / `stk_inventory.id`**，须与 201 出库扣减、401/301 退回使用的批次 **同一链路**。
- 计费行与退费行关联时，退费侧 401/301 必须使用正向 101/201 的 **同一库存行 ID**（或明确的一对一回溯字段），禁止重新选批次。

### 4.5 建议新增追溯表（设计层，待开发）

```
高值消耗明细 (dept_batch_consume_entry)
    ↓
即入即出链路表 (建议新建 gz_instant_io_link)
    ↓
stk_io_bill / stk_io_bill_entry (101|201|401|301)
    ↓
stk_inventory / stk_dep_inventory (+ 院内码、主辅条码快照)
```

字段建议见 **附录 A.1.1**（含 `linkVersion`、`writeOffCount`、`isCurrent`、`prevLinkId`、`idempotencyKey` 等）；链路明细侧另需：`stk_io_bill_id`、`stk_io_entry_id`、`inventory_id`、`dep_inventory_id`、`in_hospital_code`、`main_barcode`、`sub_barcode`、`io_direction`。

## 5. 端到端业务流程（整理版）

### 5.1 计费（正向）

```
HIS 计费镜像（待处理）
    → 科室手工高值消耗（扫码，正数）
    → 【未审核】出现在即入即出列表
    → 用户勾选 + 选仓库 + 审核
    → 自动生成 101 + 201 并自动审核
    → 仓库/科室库存流水可追溯，余额均为 0
    → 【已审核】
```

### 5.2 退费（逆向）

```
HIS 退费镜像
    → 科室手工高值反消耗（负数消耗单）
    → 【未审核】即入即出列表
    → 审核
    → 自动生成 401 + 301 并自动审核（批次/库存 ID 与原正向链路匹配）
    → 【已审核】
```

## 6. 纠错冲销

> **术语建议**：本处「冲销」指 **即入即出纠错**（备错仓库/供应商/科室等），与「HIS 镜像冲销恢复待处理」区分。

### 6.1 冲销目标

将相关 **计费/退费处理状态** 回退为可重新处理：冲销发生时需 **联动关联的退费记录**，对计费与退费两侧生成必要的反向单据/反向消耗，并将计费、退费记录均还原为 **未处理** 状态，同时撤销错误的账务链路。

### 6.2 冲销规则矩阵

#### 场景 A：高值 **计费** 记录（正向）

| 高值消耗单 | 即入即出 101+201 | 冲销动作 |
|:----------:|:----------------:|:---------|
| 已生成     | **未**生成       | 仅生成 **高值反消耗单**（撤销高值消耗） |
| 已生成     | **已**生成       | 生成 **高值反消耗单** + **退库单 401** + **退货单 301**（严格按原批次/库存 ID） |

#### 场景 B：高值 **退费** 记录（逆向）

| 高值反消耗单 | 即入即出 401+301 | 冲销动作 |
|:------------:|:----------------:|:---------|
| 已生成       | **未**生成       | 仅生成 **高值正消耗单**（恢复高值消耗态） |
| 已生成       | **已**生成       | 生成 **高值正消耗单** + **出库单 201** + **入库单 101**（严格按原链路） |

### 6.3 冲销约束

- 退库/退货/反向单据 **禁止** 手工指定与原始即入即出不一致的批次。
- 冲销须记录：操作人、时间、原因（必填）。
- 若仓库已无对应库存可供 401/301 冲销路径执行，需定义 **失败策略**（阻断 / 允许负库存告警 — 建议阻断）。
- **冲销后再次处理**：已冲销（`WRITTEN_OFF`）的记录，允许后续重新生成新的高值消耗/反消耗并再次执行即入即出，但必须满足：
  1) 新生成的即入即出链路必须使用 **新的 `instantIoLinkId`**（与历史链路严格区分）；  
  2) 同一 `gzConsumeEntryId` 在未冲销前禁止重复审核生成多套单据；冲销成功后才允许再次审核；  
  3) 冲销后的再次审核产生的新单据，必须通过“来源主键 + 链路版本（或冲销次数）”实现幂等，避免重复点击生成重复单据。

4. **科室库存为 0 的实现**：先生成并审核 **入库单 101**（生成仓库库存），再生成并审核 **出库单 201**（科室库存直接生成 qty=0 的库存行）；单据明细与库存明细均需落**院内码/主条码/辅条码**。
5. **冲销与镜像状态**：冲销需联动关联退费记录，对计费与退费两侧生成反向数据，并将计费与退费记录均还原为 **未处理** 状态。
6. **财务口径**：参与财务核算与统计，属于仓库与科室正常进出项。

### A.1.1 链路版本与冲销次数（建议表字段，便于冲销后再次即入即出）

建议新建表 `gz_instant_io_link`（名称可调整），在 **同一 `gzConsumeEntryId`** 下允许多条链路记录，用版本字段区分「当前有效链路」与「历史已冲销链路」。

| 字段名（建议） | 类型（建议） | 含义 | 使用规则 |
|----------------|--------------|------|----------|
| `instantIoLinkId` | varchar(36) | 链路主键 UUID | 每条链路唯一；冲销后再审核必须 **新生成**，禁止复用旧 ID |

| `writeOffCount` | int | 累计冲销次数 | 每执行一次纠错冲销 +1；用于审计与界面展示「第 N 次冲销」 |
| `rebuildSeq` | int | 冲销后重建序号 | 与 `linkVersion` 可二选一或并存：**推荐** `linkVersion` 表示「第几次完整即入即出」，`writeOffCount` 表示「冲销次数」；`rebuildSeq` = 冲销后再次审核成功次数（0=从未冲销后重建，1=冲销后第 1 次重建） |
| `linkStatus` | varchar(24) | 链路状态 | 与附录 B 对齐：`PENDING` / `OUT_201_DONE` / `RTH_301_DONE` / `FAILED` / `WRITTEN_OFF` |

| `prevLinkId` | varchar(36) | 上一版链路 ID | 冲销后重建时指向被冲销的 `instantIoLinkId`，便于追溯「纠错前是哪套单据」 |
| `idempotencyKey` | varchar(128) | 幂等键 | 建议格式：`{tenantId}:audit:{gzConsumeEntryId}:v{linkVersion}` 或 `{tenantId}:audit:{gzConsumeEntryId}:{linkVersion}`；同一键重复提交返回已有结果 |
| `writeOffIdempotencyKey` | varchar(128) | 冲销幂等键 | 建议格式：`{tenantId}:writeOff:{instantIoLinkId}`；防止重复冲销 |

**命名约定（开发统一口径）**

- **`linkVersion`**：强调「第几套完整即入即出链路」，用于关联 101/201/401/301 与库存 ID 的 **版本隔离**。
- **`writeOffCount`**：强调「被纠错冲销过几次」，不参与库存匹配，仅审计。
- **`rebuildSeq`**（可选）：若业务需要区分「同一镜像行冲销后重建了几次」，可与 `linkVersion` 同步递增，或仅在冲销后首次重建时 +1。

**唯一性约束（建议）**

```text
UNIQUE (tenant_id, gz_consume_entry_id, link_version)
UNIQUE (tenant_id, idempotency_key)   -- 审核幂等
PARTIAL UNIQUE: (tenant_id, gz_consume_entry_id) WHERE is_current = 1
```

4) 即入即出再次审核：生成链路 v2（linkVersion=2 或 rebuildSeq+1），新 idempotencyKey，isCurrent=1

## 附录 B：状态机与可操作矩阵（文字版）

> 目的：把“未审核/已审核/全部”的判定、半成品、失败重试、冲销联动一次性定义清楚，避免实现后界面归类与数据矛盾。

### B.1 即入即出处理状态（建议）

以“高值消耗明细（gzConsumeEntryId）”为状态载体（强一致：不允许出现可见的半成品状态）：

- **PENDING**：未做即入即出（未生成任何低值单据）
- **OUT_201_DONE**：已生成并审核出库单 201（正向完成；科室库存 qty=0 行已生成）
- **RETURN_401_DONE**：已生成并审核退库单 401（逆向处理中）
- **RTH_301_DONE**：已生成并审核退货单 301（逆向完成）
- **FAILED**：生成或审核失败（需记录 `failReason`；允许重试）
- **WRITTEN_OFF**：已冲销（并已将计费/退费镜像记录还原为未处理）

### B.2 Tab 判定规则

- **未审核 Tab**：`PENDING`、`FAILED`（以及实现上若存在半成品，也可临时放入未审核并提示状态）
- **已审核 Tab**：`OUT_201_DONE`、`RTH_301_DONE`（正向或逆向完成）
- **全部 Tab**：全部状态

> **定稿补充（强一致）**：不允许出现“仅 101 成功但 201 失败”等半成品可见状态；审核动作应在事务内保证“要么 101+201（或 401+301）全成功并审核完成，要么整体回滚并保持为 PENDING/FAILED”。

### B.3 可操作矩阵（建议）

| 状态 | 允许操作 | 说明 |
|---|---|---|
| PENDING | 审核（执行即入即出） | 弹窗选择目标仓库；按同科室+同供应商合单 |
| OUT_201_DONE / RTH_301_DONE | 冲销 | 必须联动关联退费/计费记录并回退为未处理 |
| FAILED | 重试审核 / 查看失败原因 | 必须幂等（同一来源不可重复生成多套单据） |
| WRITTEN_OFF | 查看链路 / 再次审核（新链路） | 允许再次执行即入即出生成新链路；新链路必须与历史链路版本隔离，且幂等保护生效 |

### B.4 幂等与重复点击保护（验收建议）

1. 同一 `gzConsumeEntryId` + 同一 `linkVersion`（或同一 `idempotencyKey`）的 **审核** 只能成功生成一套单据；重复点击应返回已存在结果（或提示处理中）。
2. 冲销若已执行成功（链路已为 `WRITTEN_OFF`），再次冲销应提示“已冲销”并保持数据不变。
3. 冲销后允许再次审核，但必须使用 **新的 `linkVersion` / 新的 `idempotencyKey` / 新的 `instantIoLinkId`**，且将旧链路 `isCurrent=0`。

## 附录 C：冲销思路与主流做法对齐说明（对照评估）

> 目的：对照同类系统（ERP/WMS/耗材系统）主流“红冲/冲销”处理方式，确认本文档冲销思路是否偏离主流，并给出对齐建议。

### C.1 主流系统的通用原则（总结）

1. **不改历史，用反向单据冲销**  
   原单据与原库存变更不直接删除；通过生成“反向单据/冲销单”抵消原影响，并保留审计链路。

2. **反向必须硬引用原明细/原批次/原库存行**  
   退库/退货/冲销时严格引用原出库/入库的明细与库存行（或批次唯一键），避免“重新挑批次”导致负库存与对账不一致。

3. **幂等与防重复**  
   同一来源业务主键（如一条消耗明细）在同一动作类型下只能生成一套单据；重复点击返回同一结果。

4. **状态机明确、失败可重试**  
   常见状态：待处理/已完成/已冲销/失败；并记录失败原因与重试入口。

5. **冲销后允许重做，但必须版本隔离**  
   冲销旧链路后可重新生成新链路，主流做法会用“版本号/重建序号”区分新旧链路，旧链路只读保留。

### C.2 与本文档一致的点（已对齐主流）

- **反向单据冲销而非改历史**：冲销生成反向数据（401/301 或 201/101）并联动高值正/反消耗，符合主流。
- **批次/库存一一对应**：要求 401/301 必须回溯正向同一 `stkInventoryId`，属于主流“硬引用”。
- **幂等与版本隔离**：文档已引入 `instantIoLinkId + linkVersion + idempotencyKey + prevLinkId + isCurrent`，与主流“版本化重建”一致。
- **强一致避免半成品**：属于主流系统的“更保守实现”，库存安全性更好（只要事务边界做正确）。

### C.3 主要差异点与风险（需在实现中显式兜底）

1. **科室库存 qty=0 的做法偏“轻量追溯”**  
   主流更常见的是“出库形成科室库存（数量变动）→ 再退库/领用扣减”形成数量闭环。  
   本需求为了降低维护成本，选择“科室库存直接生成 qty=0 行”。该做法并非不可行，但必须确保：
   - 401 退库/301 退货的回溯 **不依赖科室库存可用量**，而依赖 `instantIoLink`/`stkInventoryId` 的硬引用链路；
   - 所有库存/对账/报表若默认按科室库存数量变动统计，需要在实现中避免被 qty=0 误导（例如以单据明细/仓库流水为准）。

2. **强一致不允许半成品对可用性要求更高**  
   主流系统有时允许分步成功（可续跑），你们定稿为强一致回滚，意味着：
   - 101/201（或401/301）必须同事务成功，否则全部回滚；
   - 需要清晰的失败原因与重试入口，避免用户无法处理“偶发失败”。

### C.4 建议写入验收标准的“主流对齐三条红线”

1. **任何冲销/退库/退货明细必须硬引用原链路的 `stkInventoryId`（或等价主键）**，禁止人工改批次。
2. **同一 `gzConsumeEntryId` 在未冲销前不可生成第二套即入即出单据**；冲销后再审核必须 `linkVersion` 递增且 `instantIoLinkId` 变化。
3. **不允许落半成品数据**：101+201（或401+301）必须同事务成功并自动审核，否则全部回滚并进入 FAILED（可重试）。
